\section{Conceptos de Testing}

\subsection{Oracle-Based Testing}

Un \textit{test oracle} es un mecanismo para determinar si un test ha pasado o fallado \cite{barr2015oracle}. En este trabajo, el sistema Apollo original actúa como oráculo: la reimplementación se considera correcta si produce los mismos resultados que Apollo para las mismas entradas.

\textbf{Limitación:} Este enfoque asume que Apollo es correcto. Si Apollo tiene bugs, la reimplementación los replicará.

\subsection{Testing de Caja Blanca vs Caja Negra}

\textbf{Caja negra:} Diseñar tests solo basándose en la especificación de entrada/salida, sin conocer la implementación interna.

\textbf{Caja blanca:} Diseñar tests aprovechando el conocimiento de la implementación interna: parámetros, thresholds, reglas de decisión, casos límite.

Este trabajo argumenta que para sistemas ML complejos sin especificación formal, el testing de caja blanca es más efectivo porque permite diseñar tests que exploten el conocimiento interno del sistema.

\subsection{Testing Exploratorio}

El testing exploratorio consiste en diseñar y ejecutar tests de forma simultánea, basándose en el conocimiento adquirido durante el proceso. A diferencia del testing planificado, los tests se diseñan iterativamente a medida que se comprende mejor el sistema.

Este enfoque es particularmente útil para sistemas ML donde el comportamiento correcto no está formalmente especificado.

\section{Reverse Engineering de Software}

El reverse engineering es el proceso de analizar un sistema para extraer sus especificaciones a partir del código fuente \cite{chikofsky1990reverse}.

En este trabajo se aplicaron dos técnicas:

\textbf{Análisis estático:} Examinar el código sin ejecutarlo para extraer parámetros, estructuras de datos, y lógica de decisión.

\textbf{Análisis dinámico:} Ejecutar el sistema con inputs conocidos para observar y validar el comportamiento.

\section{Componentes Técnicos}

\subsection{Intersection over Union (IoU)}

Métrica para medir solapamiento entre bounding boxes:

\begin{equation}
IoU(A, B) = \frac{|A \cap B|}{|A \cup B|}
\end{equation}

Se usa en Non-Maximum Suppression (NMS) para eliminar detecciones duplicadas.

\subsection{Algoritmo Húngaro}

El algoritmo húngaro \cite{kuhn1955hungarian} resuelve el problema de asignación óptima. En el contexto de detección de semáforos, se usa para asignar detecciones a regiones de interés conocidas (projection boxes), minimizando un costo que combina distancia espacial y confianza de detección.

\subsection{Conversión Caffe a PyTorch}

Apollo utiliza modelos en formato Caffe (framework legacy). La reimplementación requiere:

\begin{enumerate}
    \item Parsear archivos \texttt{.prototxt} (definición de arquitectura)
    \item Mapear capas Caffe a equivalentes PyTorch
    \item Cargar pesos desde archivos \texttt{.caffemodel}
    \item Validar equivalencia numérica de outputs
\end{enumerate}

\section{El Sistema Apollo}

Esta sección presenta todo lo que se conocía sobre Apollo \textbf{antes} de realizar nuestro análisis. La información proviene de documentación pública y del paper de FSE 2020 \cite{peng2020first}.

\subsection{Visión General}

Apollo es una plataforma open source para vehículos autónomos desarrollada por Baidu \cite{apollo_auto}. El sistema incluye módulos para:

\begin{itemize}
    \item \textbf{Percepción:} Detección de obstáculos, semáforos, carriles
    \item \textbf{Predicción:} Predicción de trayectorias de otros vehículos
    \item \textbf{Planificación:} Planificación de ruta y maniobras
    \item \textbf{Control:} Control de actuadores del vehículo
\end{itemize}

\subsection{Módulo de Detección de Semáforos (TLR)}

Según la documentación de Apollo, el módulo TLR es responsable de:

\begin{itemize}
    \item Detectar la presencia de semáforos en imágenes de cámara
    \item Reconocer el estado actual (rojo, amarillo, verde, apagado)
    \item Mantener tracking temporal de los semáforos
\end{itemize}

\subsubsection{Pipeline Documentado}

La documentación de Apollo describe el siguiente pipeline:

\begin{enumerate}
    \item \textbf{Preprocessing:} Obtención de regiones de interés (projection boxes) a partir del HD-Map
    \item \textbf{Detection:} Red neuronal para detectar semáforos dentro de las regiones
    \item \textbf{Recognition:} Clasificadores para determinar el color del semáforo
    \item \textbf{Tracking:} Mantenimiento de estados a lo largo del tiempo
\end{enumerate}

\subsubsection{Dependencia del HD-Map}

Un aspecto importante documentado es que Apollo depende de mapas HD (High Definition Maps) para:

\begin{itemize}
    \item Conocer la ubicación 3D de semáforos en el mundo real
    \item Proyectar estas ubicaciones a coordenadas 2D en la imagen de cámara
    \item Identificar persistentemente cada semáforo mediante un ID único
\end{itemize}

Esto significa que Apollo no busca semáforos en toda la imagen, sino solo en regiones donde el HD-Map indica que debería haber semáforos.

\subsection{Arquitectura de Software}

Apollo utiliza:

\begin{itemize}
    \item \textbf{CyberRT:} Framework de mensajería propio para comunicación entre módulos
    \item \textbf{Bazel:} Sistema de build
    \item \textbf{Docker:} Containerización para deployment
    \item \textbf{Caffe:} Framework de deep learning para los modelos
\end{itemize}

\subsection{Ubicación del Código TLR}

El módulo TLR en Apollo está dividido en 4 carpetas dentro de \texttt{modules/perception/}:

\begin{verbatim}
modules/perception/
├── traffic_light_region_proposal/  # Generación de ROIs desde HD-Map
├── traffic_light_detection/        # Detección de semáforos en imagen
├── traffic_light_recognition/      # Clasificación de color
└── traffic_light_tracking/         # Tracking temporal y reglas
\end{verbatim}

Cada módulo tiene su propia estructura con archivos de configuración, código fuente, y tests.

\subsection{Paper de FSE 2020}

El paper ``A First Look at the Integration of Machine Learning Models in Complex Autonomous Driving Systems'' \cite{peng2020first} analiza cómo se integran los modelos de ML en Apollo e identifica:

\begin{itemize}
    \item \textbf{Acoplamiento estrecho:} Los componentes están fuertemente integrados
    \item \textbf{Dependencias de hardware:} Muchos componentes requieren GPUs específicas
    \item \textbf{Complejidad de configuración:} El despliegue requiere configuraciones extensas
\end{itemize}

Este paper motivó la idea de que extraer y reimplementar un subsistema podría facilitar su testing.

\section{Lo No Documentado}

Aunque la documentación de Apollo describe el pipeline a alto nivel, varios aspectos críticos \textbf{no están documentados}:

\begin{itemize}
    \item Valores específicos de parámetros (thresholds, pesos, sigmas)
    \item Reglas de seguridad del tracking temporal
    \item Lógica de decisión en casos ambiguos
    \item Justificación de decisiones de diseño
\end{itemize}

Estos aspectos solo se pueden descubrir mediante análisis del código fuente, lo cual es parte del trabajo de esta tesis.
