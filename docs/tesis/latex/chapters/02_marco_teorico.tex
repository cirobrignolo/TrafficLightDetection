\section{Oracle-Based Testing}

\subsection{Concepto de Test Oracle}

Un \textit{test oracle} es un mecanismo para determinar si un test ha pasado o fallado \cite{barr2015oracle}. En oracle-based testing, se compara el comportamiento del sistema bajo test con un oráculo de referencia que define el comportamiento correcto esperado.

\subsection{Oráculos de Implementación de Referencia}

Este trabajo utiliza un \textit{oráculo de implementación de referencia}: el sistema Apollo original actúa como referencia del comportamiento correcto. La reimplementación en PyTorch se valida comparándola directamente con Apollo.

\textbf{Ventajas de este enfoque:}
\begin{itemize}
    \item No requiere especificación formal completa
    \item La implementación de referencia es ejecutable y observable
    \item Permite validación exhaustiva mediante comparación directa de outputs
\end{itemize}

\textbf{Limitación principal:} Asume que la implementación de referencia (Apollo) es correcta. Si Apollo tiene bugs, la reimplementación los replicará.

\subsection{El Oracle Problem}

El \textit{oracle problem} \cite{barr2015oracle} se refiere a la dificultad de determinar la correctitud de las salidas cuando:

\begin{itemize}
    \item No existe especificación formal completa del comportamiento
    \item La salida correcta depende de factores complejos o inobservables
    \item Múltiples salidas pueden ser consideradas correctas
    \item El espacio de salidas es muy grande
\end{itemize}

En sistemas de ML como el detector de semáforos, el oracle problem es particularmente relevante porque:
\begin{itemize}
    \item El comportamiento correcto puede depender de factores sutiles de la imagen
    \item Los modelos neuronales son cajas negras sin especificación explícita
    \item Hay casos ambiguos donde múltiples clasificaciones son razonables
\end{itemize}

Usar Apollo como oráculo resuelve parcialmente este problema, pero introduce la limitación mencionada de asumir su correctitud.

\section{Reverse Engineering de Software}

\subsection{Definición}

El reverse engineering es el proceso de analizar un sistema para identificar sus componentes y sus interrelaciones, y crear representaciones del sistema en otra forma o a un nivel de abstracción superior \cite{chikofsky1990reverse}.

En este trabajo, el reverse engineering se aplicó para extraer las especificaciones funcionales del módulo TLR de Apollo a partir de su código fuente en C++.

\subsection{Proceso Aplicado}

El proceso de reverse engineering siguió estas etapas:

\subsubsection{Análisis Estático del Código}

Examen del código fuente sin ejecutarlo:
\begin{itemize}
    \item Identificación de estructuras de datos (e.g., \texttt{SemanticTable} para tracking)
    \item Mapeo de dependencias entre módulos
    \item Documentación de interfaces entre componentes
    \item Extracción de constantes numéricas (thresholds, pesos, etc.)
\end{itemize}

\subsubsection{Análisis Dinámico}

Observación del comportamiento durante la ejecución:
\begin{itemize}
    \item Instrumentación del código con logging
    \item Ejecución con inputs conocidos
    \item Verificación del flujo de datos entre etapas
    \item Confirmación de parámetros extraídos estáticamente
\end{itemize}

\textbf{Hallazgo clave:} Muchas decisiones de diseño (e.g., por qué NMS threshold = 0.6) no estaban documentadas. El análisis dinámico permitió validar que estos valores se usan consistentemente pero no reveló su justificación.

\subsubsection{Documentación Extraída}

El resultado fue una especificación funcional completa:
\begin{itemize}
    \item Tabla de todos los parámetros numéricos con sus valores exactos
    \item Descripción de la lógica de cada etapa del pipeline
    \item Reglas de tracking temporal con sus condiciones
    \item Casos especiales y edge cases manejados
\end{itemize}

Esta especificación sirvió como base para la reimplementación.

\section{Testing de Sistemas de Machine Learning}

\subsection{Desafíos Específicos}

El testing de sistemas ML presenta desafíos que no existen en software tradicional \cite{zhang2020machine}:

\begin{enumerate}
    \item \textbf{Ausencia de especificación formal:} El comportamiento se aprende de datos, no se programa explícitamente

    \item \textbf{Comportamiento no determinístico:} Variaciones pequeñas en input pueden cambiar outputs

    \item \textbf{Dependencia de los datos:} El comportamiento depende críticamente del dataset de entrenamiento

    \item \textbf{Degradación silenciosa:} Los modelos pueden degradarse sin errores explícitos

    \item \textbf{Dificultad para caracterizar correctitud:} ¿Qué significa que una detección sea "correcta"?
\end{enumerate}

Estos desafíos motivaron el diseño de múltiples estrategias de testing complementarias en este trabajo.

\subsection{Estrategias de Testing para Sistemas ML}

El testing de sistemas ML puede abordarse desde múltiples perspectivas complementarias:

\subsubsection{Metamorphic Testing}

El metamorphic testing \cite{chen2018metamorphic} verifica relaciones entre ejecuciones con entradas relacionadas, sin necesidad de conocer la salida correcta.

\textbf{Principio:} En lugar de preguntar "¿es este output correcto?", se pregunta "¿cómo debería cambiar el output cuando transformo el input de cierta manera?"

\textbf{Ejemplos de relaciones metamórficas:}
\begin{itemize}
    \item Invarianza al orden: Permutar inputs no debe cambiar el resultado final
    \item Escalamiento proporcional: Escalar imagen y coordenadas debe producir detecciones escaladas proporcionalmente
    \item Robustez a perturbaciones pequeñas: Ruido gaussiano pequeño no debería cambiar clasificaciones
\end{itemize}

\textbf{Ventaja:} No requiere oráculo para la salida específica, solo para relaciones entre salidas.

\subsubsection{Property-Based Testing}

Verificar que el sistema cumple propiedades invariantes independientemente del input específico.

\textbf{Ejemplos de propiedades para sistemas de detección:}
\begin{itemize}
    \item \textit{Monotonía en NMS:} Si IoU entre dos detecciones supera un threshold, solo debe mantenerse la de mayor score
    \item \textit{Conservación temporal:} Un semáforo no puede cambiar directamente de rojo a verde sin pasar por amarillo
    \item \textit{Consistencia temporal:} Los cambios de estado deben respetar tiempos mínimos físicamente plausibles
    \item \textit{Safety:} En caso de ambigüedad, sesgo hacia el estado más conservador
\end{itemize}

\subsubsection{Exploratory Testing}

Diseño manual de casos de test basado en el conocimiento del sistema para encontrar debilidades:

\textbf{Enfoque:} A partir del entendimiento profundo del código y la arquitectura, diseñar inputs específicos que puedan revelar comportamientos inesperados.

\textbf{Ejemplos de escenarios a explorar:}
\begin{itemize}
    \item Condiciones extremas de iluminación (sobreexposición, subexposición)
    \item Oclusión parcial de objetos relevantes
    \item Objetos con apariencia similar que puedan confundir al clasificador
    \item Inputs con valores en los límites de validez (timestamps negativos, coordenadas fuera de rango)
\end{itemize}

\section{Componentes Técnicos del Sistema}

Esta sección describe brevemente los componentes técnicos utilizados en el sistema, necesarios para entender la implementación pero sin entrar en detalles algorítmicos exhaustivos.

\subsection{Detección de Objetos y Métricas}

\subsubsection{Intersection over Union (IoU)}

Métrica fundamental para medir solapamiento entre bounding boxes:

\begin{equation}
IoU(A, B) = \frac{|A \cap B|}{|A \cup B|}
\end{equation}

\textbf{Uso en el sistema:}
\begin{itemize}
    \item NMS (Non-Maximum Suppression): Eliminar detecciones duplicadas con IoU $>$ 0.6
    \item Validación de detecciones: Comparar bboxes predichos vs ground truth
    \item Tracking: Calcular distancia entre detección actual y predicción
\end{itemize}

\subsubsection{Métricas de Evaluación}

\textbf{Precision y Recall:}
\begin{equation}
Precision = \frac{TP}{TP + FP} \quad \quad Recall = \frac{TP}{TP + FN}
\end{equation}

\textbf{Average Precision (AP):} Área bajo la curva Precision-Recall.

Estas métricas se utilizan para evaluar el desempeño del sistema en los tests exploratorios.

\subsection{Algoritmo Húngaro para Tracking}

El algoritmo húngaro \cite{kuhn1955hungarian} resuelve el problema de asignación óptima entre detecciones actuales y tracks existentes.

\textbf{Problema:} Dadas $n$ detecciones y $m$ tracks, encontrar la asignación que minimice el costo total.

\textbf{Input:} Matriz de costos $C \in \mathbb{R}^{n \times m}$ donde:
\begin{equation}
C_{ij} = w_1 \cdot distance(d_i, t_j) + w_2 \cdot (1 - confidence(d_i))
\end{equation}

\textbf{Output:} Conjunto de pares (detección, track) con costo total mínimo.

\textbf{Complejidad:} $O(n^3)$

\textbf{Uso en Apollo:} $w_1 = 0.7$, $w_2 = 0.3$, distancia calculada con Gaussiana 2D ($\sigma = 100$).

\section{Conversión entre Frameworks de Deep Learning}

\subsection{Caffe vs PyTorch}

Apollo utiliza Caffe (framework legacy), mientras que la reimplementación usa PyTorch:

\begin{table}[H]
\centering
\begin{tabular}{|l|p{5cm}|p{5cm}|}
\hline
\textbf{Aspecto} & \textbf{Caffe} & \textbf{PyTorch} \\
\hline
Paradigma & Declarativo (define-and-run) & Imperativo (define-by-run) \\
\hline
Debugging & Difícil & Estándar de Python \\
\hline
Estado & Legacy, poco mantenido & Activamente desarrollado \\
\hline
\end{tabular}
\caption{Diferencias clave entre frameworks}
\end{table}

\subsection{Proceso de Conversión}

La conversión de modelos requiere:

\begin{enumerate}
    \item \textbf{Mapeo de arquitectura:}
    \begin{itemize}
        \item Parsear \texttt{.prototxt} de Caffe
        \item Crear capas equivalentes en PyTorch
        \item Mapear nombres de capas y parámetros
    \end{itemize}

    \item \textbf{Conversión de pesos:}
    \begin{itemize}
        \item Leer \texttt{.caffemodel} (formato binario)
        \item Convertir formato de tensores
        \item Manejar diferencias en ordenamiento de dimensiones (e.g., NCHW vs NHWC)
    \end{itemize}

    \item \textbf{Validación numérica:}
    \begin{itemize}
        \item Comparar outputs capa por capa con misma entrada
        \item Verificar diferencias < tolerancia (típicamente $10^{-6}$)
        \item Validar gradientes si el modelo se re-entrena
    \end{itemize}
\end{enumerate}

\subsection{Desafíos de la Conversión}

\textbf{Capas custom:} Apollo usa capas propias (e.g., \texttt{RPNProposalSSD}, \texttt{DFMB-PSROIAlign}) que no tienen equivalente directo en PyTorch. Estas requirieron reimplementación completa basada en el código C++ original.

\textbf{Diferencias de precisión numérica:} Aunque ambos frameworks usan float32, diferencias en el backend de operaciones pueden producir diferencias pequeñas ($\sim 10^{-7}$) que deben ser toleradas en la validación.
