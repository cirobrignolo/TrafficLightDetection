\chapter{Análisis del Sistema}
\label{cap:analisis_apollo}

Este capítulo describe el proceso de análisis del sistema Apollo que realizamos para comprender el módulo TLR y tomar decisiones informadas sobre cómo recortarlo y reimplementarlo.

\section{Preguntas de Análisis}

El análisis estuvo guiado por cuatro preguntas fundamentales:

\begin{enumerate}
    \item \textbf{¿Cuál es el alcance funcional del módulo TLR?} \\
    ¿Qué hace exactamente? ¿Cuáles son sus entradas y salidas? ¿De qué otros módulos depende?

    \item \textbf{¿Qué partes del código son esenciales y cuáles son accesorias?} \\
    ¿Qué es la lógica core del TLR y qué es infraestructura de Apollo (CyberRT, Bazel, etc.)?

    \item \textbf{¿Qué parámetros y reglas de decisión utiliza el sistema?} \\
    ¿Cuáles son los thresholds, pesos, y reglas de negocio que determinan el comportamiento?

    \item \textbf{¿Qué simplificaciones podemos hacer sin afectar la lógica core?} \\
    ¿Qué dependencias podemos eliminar manteniendo equivalencia funcional?
\end{enumerate}

\section{Proceso de Análisis}

\subsection{Análisis Estático del Código}

Examinamos el código fuente C++ de Apollo sin ejecutarlo para extraer:

\subsubsection{Estructura del Pipeline}

Identificamos 5 etapas principales:

\begin{enumerate}
    \item \textbf{Preprocessing:} Expansión de ROIs (projection boxes → crops)
    \item \textbf{Detection:} Red CNN para detectar semáforos en los crops
    \item \textbf{Assignment:} Algoritmo húngaro para asignar detecciones a projection boxes
    \item \textbf{Recognition:} Clasificadores para determinar el color de cada detección
    \item \textbf{Tracking:} Mantenimiento de estados temporales con reglas de seguridad
\end{enumerate}

\subsubsection{Reglas de Negocio del Tracking}

Identificamos 4 reglas principales de seguridad en el código de tracking:

\textbf{Regla 1: YELLOW después de RED → Mantener RED}

\begin{codesnippet}
\begin{verbatim}
// semantic_decision.cc:145
if (color == YELLOW && iter->color == RED) {
    ReviseLights(lights, iter->color);
    // Mantiene RED por seguridad
}
\end{verbatim}
\end{codesnippet}

Justificación en comentario del código: ``Yellow only exists after green and before red. Any yellow after red is reset to red for the sake of safety.''

\textbf{Regla 2: BLACK cuando estaba encendido → Mantener color anterior}

El sistema mantiene el último color conocido cuando detecta BLACK, asumiendo que es una detección errónea temporal.

\textbf{Regla 3: Detección de Blink}

El sistema detecta parpadeo solo en verde: patrón BRIGHT → DARK ($>$0.55s) → BRIGHT.

\textbf{Regla 4: Reset después de 1.5s}

Si pasan más de 1.5 segundos sin ver un semáforo, se resetea su historial.

\section{Hallazgos del Análisis}

\subsection{Hallazgo 1: Dos Tipos de IDs}

Descubrimos que Apollo maneja dos tipos distintos de identificadores:

\begin{table}[H]
\centering
\begin{tabular}{|l|p{5cm}|p{5cm}|}
\hline
\textbf{Tipo} & \textbf{Origen} & \textbf{Persistencia} \\
\hline
\textbf{Signal ID} & HD-Map (coordenadas 3D del mundo) & Permanente \\
\hline
\textbf{Detection ID} & Asignado durante detección & Solo válido para el frame actual \\
\hline
\end{tabular}
\caption{Diferencias entre tipos de IDs}
\end{table}

\textbf{Implicación:} El tracking se basa en el Signal ID, no en la posición en píxeles. Esto significa que mover un semáforo en la imagen no confunde al sistema porque la identidad viene del HD-Map.

\subsection{Hallazgo 2: Semantic ID No Implementado}

Durante el análisis encontramos código diseñado pero no implementado:

\begin{codesnippet}
\begin{verbatim}
// traffic_light/region_proposal_component.cc:335
int cur_semantic = 0;  // Siempre 0, hardcodeado
for (auto& light : lights) {
    light->semantic = cur_semantic;
}
\end{verbatim}
\end{codesnippet}

El semantic\_id debería agrupar semáforos de la misma intersección para aplicar voting, pero está siempre asignado a 0. El código de voting existe pero nunca se ejecuta.

\textbf{Implicación:} Todos los semáforos se tratan independientemente, sin corrección por consistencia entre semáforos del mismo cruce.

\subsection{Hallazgo 3: Dependencia Crítica del HD-Map}

El HD-Map no solo provee las projection boxes, sino que es fundamental para:

\begin{itemize}
    \item Identificar persistentemente cada semáforo (Signal ID)
    \item Aplicar reglas de tracking correctamente
    \item Limitar la búsqueda a regiones conocidas
\end{itemize}

\textbf{Implicación para el recorte:} No podemos reimplementar el HD-Map dinámico, pero podemos simular su función usando projection boxes pre-calculadas manualmente.

\section{Decisiones Informadas por el Análisis}

Los hallazgos del análisis informaron las siguientes decisiones de recorte:

\subsection{Alcance Funcional}

\textbf{Incluir:}
\begin{itemize}
    \item Las 5 etapas del pipeline (preprocessing, detection, assignment, recognition, tracking)
    \item Los 4 modelos de red neuronal (detector + 3 clasificadores)
    \item Todas las reglas de tracking temporal
    \item Todos los parámetros numéricos extraídos
\end{itemize}

\textbf{Excluir:}
\begin{itemize}
    \item CyberRT (framework de mensajería)
    \item Integración con HD-Map dinámico
    \item Integración con otros módulos de Apollo
    \item Infraestructura de Docker/Bazel
\end{itemize}

\subsection{Decisiones de Interfaz}

\textbf{Entrada original (Apollo):}
\begin{itemize}
    \item Imagen de cámara (vía CyberRT)
    \item Projection boxes del HD-Map (dinámicas, calculadas en tiempo real)
\end{itemize}

\textbf{Entrada simplificada (reimplementación):}
\begin{itemize}
    \item Imagen de cámara (archivo o frame de video)
    \item Projection boxes pre-calculadas (archivo de texto)
\end{itemize}

\textbf{Justificación:} Desde el punto de vista del módulo TLR, ambas interfaces producen el mismo input: una lista de bounding boxes donde buscar semáforos. La simplificación no afecta la lógica core.

\textbf{Salida:} Se mantiene igual: lista de detecciones con bounding boxes, colores, y IDs de tracking.

\subsection{Criterio de Conformidad}

Definimos que el recorte es ``equivalente'' al original si:

\begin{enumerate}
    \item \textbf{Mismos parámetros:} Todos los valores numéricos coinciden exactamente
    \item \textbf{Misma lógica:} El código implementa exactamente las mismas operaciones en el mismo orden
    \item \textbf{Mismas reglas:} Las reglas de tracking y seguridad se aplican idénticamente
    \item \textbf{Mismos edge cases:} Los casos límite se manejan de la misma manera
\end{enumerate}

\textbf{Nota importante:} No podemos validar la equivalencia ejecutando ambos sistemas en paralelo debido a la complejidad de configurar Apollo. La equivalencia se garantiza ``por construcción'': reimplementando directamente del código fuente.

\section{Resumen}

El análisis del sistema Apollo reveló:

\begin{enumerate}
    \item \textbf{Estructura del pipeline:} 5 etapas claramente definidas
    \item \textbf{Parámetros:} Todos los valores numéricos necesarios para reimplementar
    \item \textbf{Reglas de negocio:} Lógica de tracking y seguridad
    \item \textbf{Decisiones de diseño:} Dos tipos de IDs, semantic\_id no implementado
\end{enumerate}

Este conocimiento permitió definir:

\begin{itemize}
    \item El alcance funcional del recorte
    \item Las simplificaciones de interfaz aceptables
    \item El criterio para validar equivalencia
\end{itemize}
