\chapter{Conclusiones y Trabajo Futuro}
\label{cap:conclusiones}

\section{Conclusiones}

\subsection{Sobre la Metodología}

\textbf{Reimplementar es más viable que recortar.} Para sistemas legacy complejos como Apollo, intentar recortar un módulo del sistema completo resultó inviable debido a las dependencias profundas (CyberRT, Docker, Bazel). Reimplementar desde cero, guiado por el análisis del código original, fue más efectivo.

\textbf{La reimplementación fuerza comprensión profunda.} El proceso de traducir código C++ a Python obligó a entender cada detalle del sistema: parámetros, reglas de negocio, casos especiales. Esta comprensión fue esencial para diseñar tests efectivos.

\subsection{Sobre el Testing}

\textbf{El conocimiento interno permite encontrar adversarios.} El test de transición verde→amarillo surgió del conocimiento de la arquitectura del clasificador y las clases de salida. Un enfoque de caja negra difícilmente habría encontrado el umbral exacto de falla de forma sistemática.

\textbf{El testing iterativo es natural para sistemas ML.} Sin especificaciones formales, el comportamiento correcto se descubre experimentando con el sistema. El enfoque implementar→ejecutar→observar→corregir fue más efectivo que diseñar tests exhaustivos a priori.

\subsection{Sobre el Sistema Apollo}

\textbf{Decisiones de diseño no documentadas.} El análisis reveló aspectos no documentados oficialmente: ponderación 70/30 del algoritmo húngaro, ventana temporal de 1.5\,s del \textit{tracking} con histéresis, y restricción del \textit{blink detection} al color verde con umbral de 0.55\,s. La secuencia de seguridad GREEN→YELLOW→RED, en cambio, sí está documentada \cite{apollo_tlr_docs}; lo que el análisis aportó es el conjunto completo de reglas que la rodean en la implementación.

\textbf{Funcionalidad diseñada pero no implementada.} El semantic\_id y el voting entre semáforos del mismo cruce existen en el código pero nunca se ejecutan.

\section{Contribuciones}

\begin{enumerate}
    \item \textbf{Metodología:} Demostración de que reimplementar + testing de caja blanca es viable para encontrar debilidades en sistemas ML complejos

    \item \textbf{Análisis de Apollo:} Documentación de parámetros, reglas, y decisiones de diseño no documentadas

    \item \textbf{Reimplementación:} Sistema equivalente al TLR de Apollo, simple de usar y testear

    \item \textbf{Test de transición verde→amarillo:} Caracterización del umbral de falla del clasificador (Tono -100 a -110)

    \item \textbf{Validación de hipótesis:} El conocimiento interno permite diseñar tests más efectivos que el testing de caja negra
\end{enumerate}

\section{Limitaciones}

\subsection{Del Test}

\textbf{Adversario sintético.} La modificación de color se realizó digitalmente. No se verificó si corresponde a escenarios físicamente realizables (condiciones de iluminación específicas, tipos de semáforo, etc.).

\textbf{Cobertura limitada.} Solo se exploró la frontera verde→amarillo. Otros límites de decisión no fueron testeados sistemáticamente.

\subsection{De la Reimplementación}

\textbf{Projection boxes pre-calculadas.} No se reimplementó la generación dinámica desde HD-Map, lo cual limita ciertos escenarios de testing.

\textbf{Validación por construcción.} La equivalencia con Apollo se garantizó mediante análisis de código, no mediante comparación empírica exhaustiva de outputs.

\subsection{De la Generalización}

Los hallazgos son específicos del módulo TLR de Apollo. No se puede afirmar que la metodología funcione igual para otros módulos o sistemas.

\section{Trabajo Futuro}

\subsection{Extensiones Inmediatas}

\begin{itemize}
    \item Refinar el test de transición: encontrar umbral exacto entre Tono -100 y -110
    \item Explorar otras fronteras de decisión: amarillo→rojo, verde→black
    \item Verificar si el adversario sintético corresponde a escenarios físicos reales
\end{itemize}

\subsection{Extensiones a Mediano Plazo}

\begin{itemize}
    \item Aplicar la metodología a otros módulos de Apollo (detección de peatones, carriles)
    \item Implementar el semantic\_id y voting para evaluar su impacto
    \item Crear dataset de evaluación con ground truth para métricas cuantitativas
\end{itemize}

\subsection{Direcciones de Investigación}

\begin{itemize}
    \item Formalizar el proceso de derivar tests a partir de conocimiento interno
    \item Desarrollar métricas de equivalencia entre implementaciones de sistemas ML
    \item Explorar automatización del proceso de análisis y reimplementación
\end{itemize}

\section{Reflexión Final}

Este trabajo demostró que para testear efectivamente un sistema ML complejo, a veces es necesario \textbf{focalizar}: elegir un subsistema, comprenderlo profundamente, y diseñar tests específicos basados en ese conocimiento.

La complejidad de sistemas como Apollo hace impracticable el testing exhaustivo del sistema completo. Pero si recortamos un subsistema y lo reimplementamos de forma equivalente, ganamos control y comprensión que nos permiten encontrar adversarios de forma sistemática.

La contribución principal no es la reimplementación en sí, sino la \textbf{demostración de que este enfoque funciona}: el conocimiento adquirido durante la reimplementación permitió diseñar un test que encontró el umbral exacto de falla del clasificador.
